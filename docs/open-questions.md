21. ⚠️ **SUPERSEDED by #28 (same day; jitter concept survives, salt/zones do not). Original kept for rationale:** ✅ **DECIDED (2026-07-06) — "Pin ring" mitigation: salted per-route radius jitter.** Found in review: with a fixed 300 m trim, every published pin/endpoint sits ~on a circle around home (Flow samples ~1/s ≈ every 5–6 m, so the first surviving point is within meters of the boundary); three clean pins suffice to fit the circle and recover the home location precisely, defeating the zone. Fix: each route's effective trim radius for a zone is stretched deterministically into `[radius_m, radius_max_m]` (default 300–600 m) via **HMAC(salt, filename)** — salt is secret and lives in gitignored `privacy-zones.json` (template: `privacy-zones.example.json`). Deterministic ⇒ pipeline re-runs are idempotent (no file churn); secret salt ⇒ an attacker can't reconstruct per-route radii, so circle-fitting fails. Residual (accepted): many rides starting from one area still disclose the *neighborhood*; the zone's job is doorstep-level protection. Averaging attacks remain but are blunted by jitter + road-network anisotropy. **Risk context (user, 2026-07-06): only a handful of routes originate from home — most are trips around Czechia starting/ending at various non-sensitive locations (jitter kept anyway as cheap insurance; the ring attack needs only ~3 clean pins).**
21b. ⚠️ **SUPERSEDED by #28 — under content-keyed jitter, renames are radius-irrelevant by construction and the gate is structural, so the rule below is moot. Original:** **Jitter verification rule (spec'd 2026-07-06 — prevents rename breakage):** the jittered radius is a **trim-time** choice only. The pre-commit gate and privacy tests verify the **`radius_m` floor** (default 300 m), never the per-filename jittered radius — otherwise renaming a published route (jitter is keyed on filename) would demand a re-trim from a raw file that may no longer exist. Renames therefore never touch geometry: rename in `gpx/` → regenerate `routes.json` → done.

28. ✅ **DECIDED (user, 2026-07-06) — Privacy model v2: endpoint-relative trim, zero config, no secrets.** The zones model (#17/#21/#21b/#25) is replaced. Rationale: rides originate from many "homes" (house, campsite, hotel) — a fixed zone list doesn't fit — and the user will not enter real home coordinates or hand-generate secrets anywhere, gitignored or not. New model: trim every point within a jittered radius of the ride's **own first raw point** and (independently) its **last raw point**; radii drawn from `[300 m, 600 m)` keyed on **SHA-256 of the raw file bytes** (deterministic re-runs; unreconstructable by outsiders; renames radius-irrelevant by construction). Pin = first surviving point; fully-trimmed track → skip + warn. The gate is zero-config: structural whitelist + pin-must-equal-first-trkpt consistency + deny-list replace the absolute zone check. **Accepted residuals (user-visible, stated at decision time):** a ride merely *passing* home mid-route is published in full (only start/end anchors trim); repeat rides from one area still disclose the neighborhood (jitter blunts doorstep recovery only).

## Tune later (decide before it bites)

8. **Fixtures policy (adopted, tightened 2026-07-06):** committed test GPX must be **fully synthetic geometry** (generated shapes — loops, out-and-backs — at fabricated coords). Do **not** transpose the real ride's geometry to fake coordinates: a route's *shape* can be map-matched back to the real road network and re-located. The real `raw/eBike ride.gpx` is **local dev input only**, never a committed fixture.
9. **Same-day duplicate filename.** `YYYY-MM-DD Name.gpx` collides if the same named loop is ridden twice in one day → silent overwrite. Add a disambiguator (`-2`, or a time component).
10. **Czech diacritics + NFC/NFD + spaces in filenames.** macOS stores NFD, Windows/Linux NFC → same name, different bytes across mac-dev / Win-BFU / Pages-Linux → 404s / broken permalinks. Decouple the URL **slug** from the raw filename. *Proposal (low-stakes, adopt unless objected):* slug = NFC-normalize → ASCII-fold diacritics → lowercase → non-alnum runs → `-` (e.g. `2026-07-06 Okolo Přehrady.gpx` → `2026-07-06-okolo-prehrady`); the **pipeline** computes it and stores it in `routes.json` (never derived client-side, so NFC/NFD can't bite in JS); the site fetches the GPX by URL-encoded filename also taken verbatim from `routes.json`. Additionally the **pipeline NFC-normalizes filenames on emit** (macOS writes NFD; GitHub Pages serves bytes as-committed — committing NFC from the start prevents the mismatch class entirely).
11. **`routes.json` scope (frozen, implemented 2026-07-06):** top-level `{"routes": [...]}`; per route: `slug`, `file` (exact filename for fetch), `name`, `date`, `distance_km`, `elevation_m`, `pin` [lat,lng], `bbox`. No `generated` timestamp (dropped — it churned the diff on every rebuild). Index build collects per-file errors (malformed/impossible-date filenames, slug collisions → first by sorted filename wins) and still writes the good entries with a nonzero exit, so one bad file can't block 1400 routes. No full track geometry (bloats first load); no coord derived from the raw start. *Regeneration model (proposal):* `routes.json` is **always rebuilt by scanning `gpx/`** (stats from the committed trimmed+simplified files) rather than incrementally from `raw/` — idempotent, reproducible from public data alone, and renames/deletions in `gpx/` self-heal on the next run. Accepted cost: distance/elevation come from the simplified line (slight underestimate; validate against the Flow app's stats for the real ride).
11b. **Elevation-gain algorithm.** Naïve summation of per-point `<ele>` deltas on ~1 Hz data grossly overestimates ascent (sensor noise). Use threshold/hysteresis smoothing (only count climbs once they exceed ~2–3 m). Tune by comparing the pipeline's number against what Bosch Flow reports for `raw/eBike ride.gpx`.
12. **Mapy credits (researched 2026-07-06 — see docs/mapy-api.md).** 1 credit/tile; Basic = 250k free credits/mo; **Extended tariff = 10M free/mo for public free projects — this site qualifies, apply at deploy** (conditions: public, Mapy-data-only, logo+copyright, catalogue listing, annual affidavit). No silent charges: paid overage requires explicit opt-in, otherwise tiles just stop. Still mock tiles in the automated suite except one real attribution/visual check. Preview = 10 maps/page.
13. **Simplification tolerance.** Douglas–Peucker starting value **8 m** (now stated as the decided starting value in CLAUDE.md), validate visually against `raw/eBike ride.gpx` and re-tune.
14. **Read-only agents hold Bash.** `verifier`/`grill-reviewer` are "read-only" by convention but Bash can mutate git. Consider an allowlist or explicit "never stage/commit" in their prompts.
15. **Safety-critical trim code is written by Sonnet `developer`, critiqued by Opus.** Consider having Opus co-design/write the trim+emit algorithm given how load-bearing it is.
16. **`.gitignore`/repo hygiene:** consider `.nojekyll` (plain static site); ~~add a committed `privacy-zones.example.json`~~ **done 2026-07-06** (includes the jitter salt). No-key case → graceful "map disabled — see config.example.js" message, not a hard crash.
16b. **Leaflet version + vendoring (researched 2026-07-06).** Vendor Leaflet + Leaflet.markercluster into the repo (e.g. `lib/`) instead of a CDN — no build step means no lockfile, a CDN outage/URL rot breaks the site, and the Phase 2 QtWebEngine preview reuses the same page. **Pin Leaflet 1.9.x** — Leaflet 2.0 is still pre-stable (ESM-only, global `L` removed, plugin ecosystem incl. markercluster not settled). GPX rendering: leaflet-gpx is alive (v2.2.0, 2026) but our emitted GPX is whitelist-simple — a ~30-line DOMParser→`L.polyline` renderer avoids the dependency; dev's call at scaffold time.
